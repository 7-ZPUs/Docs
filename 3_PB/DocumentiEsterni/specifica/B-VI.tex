\section{Backend - Verifica Integrita}

\subsection{Panoramica}
\label{sec:integrity}

\noindent
Il modulo Verifica Integrita esegue controlli su 5 livelli gerarchici:
File $\to$ Documento $\to$ Processo $\to$ Classe documentale $\to$ DIP.
Il controllo puo essere avviato da qualsiasi livello e propaga lo stato nel sottoalbero interessato.

\subsubsection{Perimetro funzionale}
\label{sec:integrity:perimetro}

\noindent
La verifica opera sui seguenti livelli:
\begin{enumerate}[labelwidth=0.8cm,leftmargin=1cm]
	\item \textbf{File}
	\item \textbf{Documento}
	\item \textbf{Processo}
	\item \textbf{Classe documentale}
	\item \textbf{DIP}
\end{enumerate}

\noindent
Ogni invocazione calcola lo stato del nodo root e dei suoi discendenti, con persistenza progressiva in SQLite tramite \codeword{updateIntegrityStatus(...)}.

\subsubsection{Flusso end-to-end}
\label{sec:integrity:flusso}

\paragraph{Sequenza di verifica.}
\begin{enumerate}[labelwidth=0.8cm,leftmargin=1cm]
	\item Il renderer invoca il canale \codeword{check-integrity:*}
	\item \codeword{CheckIntegrityIpcAdapter} intercetta l'handler e risolve il use case
	\item Il use case delega a \codeword{IntegrityVerificationService}
	\item Il service esegue \codeword{runInTransaction(...)}
	\item Il service verifica hash file e aggrega stati sui parent
	\item Il service persiste gli stati sui livelli coinvolti
	\item L'handler IPC ritorna \codeword{IntegrityStatusEnum}
\end{enumerate}

\paragraph{Flusso gerarchico di propagazione.}
\begin{center}
	\codeword{CHECK\_DIP\_INTEGRITY\_STATUS} \\
	$\downarrow$ \\
	\codeword{CHECK\_DOCUMENT\_CLASS\_INTEGRITY\_STATUS} \\

\subsubsection{Servizio centrale}
\label{sec:integrity:service}

\noindent
File: \codeword{IntegrityVerificationService.ts}.

\paragraph{Dipendenze iniettate.}
\begin{enumerate}[labelwidth=0.8cm,leftmargin=1cm]
	\item \codeword{IFileRepository}
	\item \codeword{IDocumentRepository}
	\item \codeword{IProcessRepository}
	\item \codeword{IDocumentClassRepository}
	\item \codeword{IDipRepository}
	\item \codeword{IHashingService}
	\item \codeword{ITransactionManager}
	\item \codeword{DIP\_PATH\_TOKEN}
\end{enumerate}

\paragraph{Regole operative.}
\begin{enumerate}[labelwidth=0.8cm,leftmargin=1cm]
	\item Ogni API pubblica gira dentro \codeword{runInTransaction}
	\item Se la root non esiste, viene lanciato errore
	\item La verifica e ricorsiva nel sottoalbero
	\item Ogni livello aggiorna \codeword{integrity\_status} via repository
\end{enumerate}

\subsubsection{Regole di aggregazione}
\label{sec:integrity:aggregation}

\noindent
\textbf{Caso 1:} nessun figlio $\to$ \codeword{VALID}\\
\textbf{Caso 2:} almeno un figlio \codeword{INVALID} $\to$ \codeword{INVALID}\\
\textbf{Caso 3:} nessun \codeword{INVALID} ma almeno un \codeword{UNKNOWN} $\to$ \codeword{UNKNOWN}\\
\textbf{Caso 4:} tutti \codeword{VALID} $\to$ \codeword{VALID}

\subsubsection{Hashing e transazioni nel modulo}
\label{sec:integrity:infra}

\paragraph{HashingService.}
\noindent
\codeword{checkFileIntegrity(filePath, expectedHash)} legge bytes, calcola SHA-256, converte digest Base64 e confronta con l'hash atteso.

\paragraph{Path assoluto file.}
\noindent
Il service risolve il percorso con \codeword{path.resolve(dipPath, file.getPath())} prima della verifica hash.

\paragraph{Transaction manager.}
\noindent
L'esecuzione avviene in \codeword{runInTransaction(work)}; in caso di errore durante la cascata e garantito rollback completo.

\subsubsection{Comportamenti runtime}
\label{sec:integrity:runtime}

\noindent
\textbf{Root non trovato:} eccezione su id non presente\\
\textbf{Persistenza:} aggiornamento \codeword{integrity\_status} su tutti i livelli coinvolti\\
\textbf{Atomicita:} rollback completo su errore interno\\
\textbf{Return IPC:} stato finale \codeword{IntegrityStatusEnum}
